## 1. 系统概述

### 1.1 设计哲学

Claude天团的技术系统遵循"现金优先，软件次之"的原则。系统的核心目标是：

- **加速收入循环**：从线索到收款的全流程提速
- **降低维护负担**：每周维护时间不超过15分钟
- **保持决策透明**：所有信息可在5分钟内审阅完毕

### 1.2 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│                    技术系统核心原则                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 服务先行        - 付费服务优先于软件产品                │
│  2. 固定范围        - 每个报价都有明确边界                   │
│  3. 静态优先        - 静态页面、Markdown、模板优先          │
│  4. 无持久化服务    - 无数据库、无后台任务、无用户系统       │
│  5. 痛点驱动        - 问题出现3次才考虑自动化               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 系统健康指标

系统健康的唯一判断标准：

| 指标 | 目标值 | 负责人 |
|------|--------|--------|
| CTO审阅时间 | < 5分钟 | CTO |
| 活跃构建发布 | ≤ 1天 | CTO |
| 每周维护时间 | < 15分钟 | CTO |
| 工程项与指标关联 | 100% | CTO |

### 2.2 各层详解

#### Layer 1: 需求捕获 (Demand Capture)

**目标**: 将流量和外联转化为对话

| 组件 | 实现方式 | 位置 |
|------|----------|------|
| 官网 | 静态HTML | `docs/` |
| 服务展示页 | 静态HTML | `docs/services.html` |
| 联系方式 | 页面内嵌 | `docs/contact.html` |
| 线索追踪 | Markdown表格 | `company/dashboard.md` |

**关键流程**:
```
访问者 → 浏览服务页 → 点击联系 → 填写表单/邮件 → CTO记录到仪表盘
```

#### Layer 2: 资格确认与报价 (Qualification and Quote)

**目标**: 将对话转化为固定范围报价

| 组件 | 实现方式 | 位置 |
|------|----------|------|
| 资格确认流程 | Markdown文档 | `company/operations/order-intake.md` |
| 标准报价包 | Markdown模板 | `company/offers/` |
| 报价生成 | 手动填充模板 | `company/templates/offer-pack-template.md` |

**报价生成流程**:
```
1. 确认客户需求类型
2. 选择匹配的服务类型
3. 使用对应报价包模板
4. 填充具体范围和价格
5. 发送给客户确认
```

#### Layer 3: 支付与订单激活 (Payment and Order Activation)

**目标**: 将接受报价转化为已付款订单

| 组件 | 实现方式 | 位置 |
|------|----------|------|
| 支付指南 | Markdown文档 | `company/operations/payment-runbook.md` |
| 支付请求 | 手动发送 | - |
| 支付确认 | 仪表盘记录 | `company/dashboard.md` |

**支付流程**:
```
报价接受 → 发送支付指南 → 客户付款 → 确认到账 → 激活订单
```

#### Layer 4: 交付与质量保证 (Delivery and QA)

**目标**: 将已付款订单转化为可预测产出

| 组件 | 实现方式 | 位置 |
|------|----------|------|
| 交付工具包 | Markdown检查清单 | `company/delivery-kits/` |
| QA检查清单 | Markdown文档 | `company/operations/delivery-quality-checklist.md` |
| 交付记录 | 仪表盘更新 | `company/dashboard.md` |

**当前支持的交付类型**:
- 代码审查服务
- 内容创作服务
- 研究报告服务

#### Layer 5: 成果证明与复用 (Proof and Reuse)

**目标**: 将已完成工作转化为促进下次销售的资产

| 组件 | 实现方式 | 位置 |
|------|----------|------|
| 案例研究模板 | Markdown模板 | `company/templates/case-study-template.md` |
| 服务协议模板 | Markdown模板 | `company/templates/service-agreement-template.md` |
| 成果资产库 | 文件归档 | `company/templates/` |

## 3. 技术边界与约束

### 3.1 当前阶段定义

**阶段 0 - 手动牵引期 (Stage 0 - Manual Traction)**

```
┌─────────────────────────────────────────────────────────────┐
│                    Stage 0 特征                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✓ 服务优先于产品                                           │
│  ✓ 固定范围报价优先于定制                                    │
│  ✓ 仅使用静态页面、Markdown、模板、本地脚本                  │
│  ✗ 无持久化服务                                             │
│  ✗ 无认证系统                                               │
│  ✗ 无数据库                                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 允许的技术范围

**当前允许**:
- 静态网站页面
- Markdown文件
- 可复用模板
- 无需托管的本地脚本
- 手动支付处理
- 仪表盘内的手动CRM纪律

**当前禁止**:
- 数据库
- 后台任务
- 登录或用户账户
- 管理后台应用
- 自定义CRM
- 自定义结账系统
- 需要监控的Webhook自动化

### 3.3 技术升级阶梯

每个重复出现的运营问题必须按以下顺序升级：

```
┌─────────────────────────────────────────────────────────────┐
│                    技术升级阶梯                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Level 1: 手动纪律                                          │
│           ↓ (问题重复出现)                                   │
│  Level 2: 可复用模板或检查清单                               │
│           ↓ (问题持续出现)                                   │
│  Level 3: 静态页面或本地脚本                                 │
│           ↓ (阶段门审批通过)                                 │
│  Level 4: 狭义软件系统                                       │
│                                                             │
│  ⚠️ 不可跳级！                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**升级触发条件**:

| 触发条件 | 允许的响应 |
|----------|------------|
| 单次摩擦 | 手动修复 |
| 同一问题出现2次 | 优化文案、模板或检查清单 |
| 同一问题7天内出现3次 | 发布本地脚本或静态页面改进 |
| 收入受阻1次 | 发布最小化的当天修复 |
| 发布后维护负担增加 | 删除或简化构建 |

## 7. 决策框架

### 7.1 技术决策过滤器

任何新构建必须通过以下过滤器：

```
                    技术决策过滤器
                    ══════════════════

问题 1: 是否直接改善五层架构之一?
        │
        ├── 否 → 拒绝
        │
        └── 是 ↓

问题 2: 是否在允许的技术范围内?
        │
        ├── 否 → 拒绝或延迟到阶段升级
        │
        └── 是 ↓

问题 3: 是否通过升级阶梯?
        │
        ├── 否 (跳级) → 返回上一级
        │
        └── 是 ↓

问题 4: 是否满足工程预算约束?
        │
        ├── 否 → 缩小范围或拒绝
        │
        └── 是 ↓

问题 5: 是否有明确的指标关联?
        │
        ├── 否 → 拒绝
        │
        └── 是 → 批准执行
```

### 7.2 工程预算约束

每周技术预算限制：

| 约束项 | 限制 |
|--------|------|
| 进行中工程项 | 最多1个 |
| 关联指标 | 必须有1个 |
| 最小可发布版本 | ≤1天 |
| 新增维护时间 | ≤15分钟/周 |

### 7.3 删除规则

删除或延迟任何：

- 需要第二个追踪器的东西
- 增加维护但不改善速度、质量或转化的东西
- 在重复付费需求存在前假设未来规模的东西
- 试图用软件解决创始人纪律问题的东西
- 无法用一句话关联一个指标解释的东西

## 8. 最佳实践

### 8.1 文档维护最佳实践

1. **保持单一真相源**: 永远不要让信息存在于两个地方
2. **定期清理**: 每周检查并删除过时信息
3. **版本控制**: 使用Git追踪所有变更
4. **明确所有权**: 每个文件都有明确的负责人

### 8.2 流程优化最佳实践

1. **先优化内容**: 页面文案优化先于代码
2. **先模板后脚本**: 模板优化先于脚本自动化
3. **先本地后托管**: 本地脚本先于托管系统
4. **先报价后工具**: 优化报价本身先于围绕报价构建工具

### 8.3 决策记录最佳实践

所有重要技术决策应记录在：
- `docs/decision-log.md` - 决策日志
- `company/dashboard.md` - 影响当前运营的决策

决策记录格式：
```markdown
## YYYY-MM-DD: 决策标题

**背景**: 为什么需要做这个决定
**选项**: 考虑了哪些选项
**决策**: 选择了什么
**理由**: 为什么这样选择
**指标**: 如何衡量成功
```

### 8.4 问题升级最佳实践

```
问题出现频率决策树:

第1次出现 → 手动解决，记录观察

第2次出现 → 评估是否需要模板/检查清单改进

第3次出现 (7天内) → 必须启动流程优化或自动化

收入受阻 (任何次数) → 当天修复，优先级最高
```

### B. 常见问题

**Q: 什么时候可以升级到下一阶段?**
A: 参考 `company/operations/technical-roadmap.md` 中的触发条件。

**Q: 如何判断一个构建是否值得?**
A: 通过决策过滤器，必须直接改善五层架构之一，且能用一句话关联一个指标。

A: 按升级阶梯处理：手动 → 模板 → 脚本 → 软件（需审批）。

**Q: 系统复杂度增加时怎么办?**
