软件架构是系统的骨架，决定了软件的质量属性：可维护性、可扩展性、性能、安全性。好的架构能让团队事半功倍，糟糕的架构则是技术债务的源头。本文总结我们在数十个项目中验证过的架构设计最佳实践。

核心设计原则

所有架构设计都应该遵循一些基本原则，这些原则跨越技术和时代：

1. 关注点分离（Separation of Concerns）

每个模块应该只负责一件事，并且把它做好。这是软件工程最基本的原则，但也是最容易被违反的。

实践要点：
  • UI层只处理展示逻辑
  • 业务层只处理业务规则
  • 数据层只处理数据存取
  • 避免在错误的地方放错误的代码

2. 高内聚低耦合

模块内部的元素应该紧密相关（高内聚），模块之间应该尽量独立（低耦合）。

// 高内聚示例：用户模块只处理用户相关逻辑 class UserService { constructor(userRepository, emailService) { this.userRepository = userRepository; this.emailService = emailService; } async createUser(userData) { const user = await this.userRepository.create(userData); await this.emailService.sendWelcomeEmail(user.email); return user; } async getUser(id) { return this.userRepository.findById(id); } } // 低耦合：通过接口依赖，而非具体实现 interface IUserRepository { create(data: UserData): Promise<User>; findById(id: string): Promise<User | null>; }

3. 单一职责原则（SRP）

一个类或模块应该有且只有一个改变的理由。

4. 依赖倒置原则（DIP）

高层模块不应依赖低层模块，两者都应依赖抽象。

5. 接口隔离原则（ISP）

不应强迫客户依赖它们不使用的方法。

架构决策记录（ADR）

架构决策需要被记录下来，否则团队会反复讨论相同的问题，新成员也无法理解决策背后的原因。

ADR模板

# ADR-001: 使用PostgreSQL作为主数据库 ## 状态 已接受 ## 背景 我们需要为电商平台选择主数据库，系统预期特点： - 日订单量：10万+ - 需要复杂查询（订单分析、用户画像） - 需要事务支持（订单支付） - 预算有限 ## 决策 选择PostgreSQL作为主数据库 ## 理由 1. **事务支持**：完整的ACID支持，适合订单系统 2. **复杂查询**：支持窗口函数、CTE等高级特性 3. **成本**：开源免费，社区活跃 4. **扩展性**：支持分区、读写分离 ## 替代方案 - MySQL：简单查询性能好，但复杂查询能力弱 - MongoDB：灵活性好，但事务支持弱 - TiDB：分布式能力强，但运维复杂度高 ## 影响 - 需要团队成员学习PostgreSQL特有语法 - 运维需要建立备份和监控体系 - 后续可考虑使用Citus进行水平扩展 ## 日期 2026-03-10
实践建议：将ADR文档放在代码仓库的`/docs/adr`目录下，使用Git版本控制。每个重大架构决策都应该有对应的ADR。

常见架构模式

分层架构（Layered Architecture）

最经典的架构模式，适合大多数业务系统。

层 职责 示例
表示层 处理用户交互 Controllers, Views
应用层 编排业务流程 Application Services
领域层 核心业务逻辑 Entities, Domain Services
基础设施层 技术支持 Repositories, External APIs

六边形架构（Hexagonal Architecture）

又称端口与适配器架构，强调核心业务与外部世界的隔离。

核心思想：
  • 领域核心：纯业务逻辑，不依赖任何框架
  • 端口：定义与外部交互的接口
  • 适配器：实现端口，连接外部系统

事件驱动架构（Event-Driven Architecture）

通过事件进行组件间通信，实现松耦合。

典型场景：
  • 用户注册 → 发送欢迎邮件
  • 订单创建 → 扣减库存
  • 支付完成 → 更新订单状态 → 发送通知

CQRS（命令查询职责分离）

将读操作和写操作分离，各自优化。

// 命令模型：负责写操作 class CreateOrderCommand { constructor( public userId: string, public items: OrderItem[], public shippingAddress: Address ) {} } class OrderCommandHandler { async handle(command: CreateOrderCommand): Promise<string> { // 写入主数据库 const order = await this.orderRepository.create(command); // 发布事件 await this.eventBus.publish(new OrderCreatedEvent(order)); return order.id; } } // 查询模型：负责读操作 class OrderQueryService { async getOrderSummary(orderId: string): Promise<OrderSummary> { // 从读库查询（可能是物化视图） return this.readDb.query('SELECT * FROM order_summaries WHERE id = ?', [orderId]); } }

架构反模式

1. 大泥球（Big Ball of Mud）

症状：系统没有清晰的架构边界，所有代码都混杂在一起。

原因：缺乏架构设计，随需求不断堆砌代码。

解决方案：识别核心领域，逐步重构，建立模块边界。

2. 分布式单体（Distributed Monolith）

症状：名义上是微服务，实际上服务间紧密耦合，部署需要协调所有服务。

原因：错误的微服务拆分，服务边界不合理。

解决方案：按业务能力拆分，服务间通过API或事件通信，避免共享数据库。

3. 过度设计（Over-Engineering）

症状：为未来可能永远不会发生的需求设计复杂的架构。

原因：过度追求完美，忽视当前业务价值。

解决方案：YAGNI原则——不要预测未来，只做当前需要的，保留扩展点。

4. 金锤子（Golden Hammer）

症状：手里有把锤子，看什么都是钉子。用同一套架构方案解决所有问题。

原因：对特定技术过度熟悉，缺乏技术广度。

解决方案：扩展技术视野，根据问题选择合适的方案。

从单体到微服务

微服务不是目的，而是手段。盲目追求微服务只会增加复杂性。

何时考虑微服务？

  • 团队规模超过20人，代码冲突频繁
  • 不同模块有不同的扩展性需求
  • 需要独立部署和发布不同模块
  • 技术栈需要多样化

何时保持单体？

  • 团队规模小（<10人）
  • 业务快速迭代，边界不清晰
  • 运维能力不足
  • 对延迟要求严格

演进路径

阶段1：单体先行

从单体开始，但保持模块化设计。在单体内部建立清晰的模块边界。

阶段2：识别边界

根据业务领域识别潜在的微服务候选。参考领域驱动设计（DDD）的限界上下文（Bounded Context）概念。

阶段3：逐步拆分

优先拆分变化最频繁、价值最高的模块。每次拆分一个服务，验证效果后再继续。

阶段4：独立部署

建立CI/CD流水线，实现服务的独立部署和扩展。

重要提醒：不要一开始就设计微服务架构。先让单体跑起来，在真正遇到瓶颈时再考虑拆分。记住：可以拆分的前提是先有一个可拆分的单体。

可扩展性设计

水平扩展 vs 垂直扩展

方式 优点 缺点 适用场景
垂直扩展 简单，无需改代码 成本增长快，有上限 数据库、缓存
水平扩展 成本线性增长，理论上无限 需要架构支持，复杂性增加 无状态服务

无状态设计

水平扩展的前提是无状态设计。

// 错误：有状态的服务 class BadOrderService { private currentOrder: Order; // 状态保存在服务内存中 addItem(item: OrderItem) { this.currentOrder.items.push(item); } } // 正确：无状态的服务 class GoodOrderService { constructor(private orderRepository: IOrderRepository) {} async addItem(orderId: string, item: OrderItem) { const order = await this.orderRepository.findById(orderId); order.items.push(item); await this.orderRepository.save(order); return order; } }

数据库扩展策略

  1. 读写分离：写主库，读从库
  2. 分库分表：按业务或数据特征拆分
  3. 缓存层：Redis缓存热点数据
  4. CDN加速：静态资源就近访问

架构设计流程

1. 需求分析

  • 功能需求：系统要做什么？
  • 非功能需求：性能、可用性、安全性等
  • 约束条件：预算、团队、时间

2. 架构设计

  • 选择架构模式
  • 定义组件和接口
  • 选择技术栈

3. 评审验证

  • 架构评审会议
  • 原型验证
  • 风险评估

4. 实施演进

  • 迭代开发
  • 持续重构
  • 架构治理

架构设计检查清单

检查项 问题
需求对齐 架构是否满足所有关键需求？
边界清晰 组件职责是否明确？边界是否清晰？
扩展性 能否应对未来增长？
可用性 单点故障如何处理？
性能 能否满足性能指标？
安全性 数据如何保护？访问如何控制？
可观测性 如何监控和诊断问题？
团队能力 团队是否有能力实施和维护？

总结

架构设计没有银弹，只有权衡。好的架构师不是选择"最好的"方案，而是选择"最合适"的方案。

关键原则：

  • 简单优于复杂：能用简单方案就不要用复杂方案
  • 演进优于预测：不要为未来过度设计，保留扩展点即可
  • 记录优于遗忘：用ADR记录架构决策及其原因
  • 验证优于假设：通过原型和测试验证架构假设

架构设计是一个持续的过程，不是一次性的活动。随着业务发展和技术演进，架构也需要不断调整和优化。
