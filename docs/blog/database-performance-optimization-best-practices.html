第一章：性能问题的诊断方法

在开始优化之前，必须先准确定位问题。盲目的优化不仅浪费资源，还可能引入新问题。

1.1 性能基准测试

建立性能基准是优化的第一步：

-- 记录查询执行时间 SET profiling = 1; SELECT * FROM orders WHERE customer_id = 12345; SHOW PROFILE; -- 查看执行计划 EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 12345 AND status = 'pending';

1.2 慢查询日志分析

慢查询日志是发现性能问题的金矿：

# MySQL 配置 slow_query_log = 1 slow_query_log_file = /var/log/mysql/slow.log long_query_time = 2 log_queries_not_using_indexes = 1 # 使用 pt-query-digest 分析 pt-query-digest /var/log/mysql/slow.log
实用技巧：不要只关注执行时间最长的查询，还要关注执行频率最高的查询。一个执行100ms但每天运行100万次的查询，比一个执行10秒但每天运行10次的查询更值得优化。

1.3 系统监控指标

指标 正常范围 异常信号
QPS (Queries Per Second) 根据硬件而定 突然下降
连接数 < max_connections的80% 接近上限
缓冲池命中率 > 95% < 90%
锁等待时间 < 100ms > 1s
临时表创建数 尽量少 大量磁盘临时表

第二章：索引优化 - 性能优化的核心

索引是数据库性能优化的基石。正确的索引设计可以带来数量级的性能提升。

2.1 索引类型详解

B+Tree 索引

最常用的索引类型，适用于范围查询、排序和分组：

-- 创建B+Tree索引 CREATE INDEX idx_user_email ON users(email); CREATE INDEX idx_order_user_status ON orders(user_id, status); -- 适用场景 SELECT * FROM users WHERE email = 'test@example.com'; SELECT * FROM orders WHERE user_id = 123 AND status = 'pending'; SELECT * FROM orders WHERE user_id = 123 ORDER BY created_at;

哈希索引

只支持等值查询，不支持范围查询和排序：

-- MySQL Memory引擎支持哈希索引 CREATE TABLE sessions ( id INT PRIMARY KEY, session_hash VARCHAR(64), INDEX USING HASH (session_hash) );

全文索引

用于文本搜索：

-- 创建全文索引 CREATE FULLTEXT INDEX idx_content ON articles(title, content); -- 使用全文搜索 SELECT * FROM articles WHERE MATCH(title, content) AGAINST('数据库优化' IN NATURAL LANGUAGE MODE);

2.2 复合索引设计原则

复合索引的设计需要遵循"最左前缀原则"：

示例：复合索引设计

假设有一个订单表，经常有以下查询：

  • WHERE user_id = ? AND status = ?
  • WHERE user_id = ? AND created_at > ?
  • WHERE user_id = ? ORDER BY created_at DESC

最优索引设计：

-- 索引顺序：user_id → status → created_at CREATE INDEX idx_order_query ON orders(user_id, status, created_at); -- 这个索引可以覆盖： -- 1. WHERE user_id = ? AND status = ? (使用前两列) -- 2. WHERE user_id = ? AND status = ? AND created_at > ? (使用全部三列) -- 3. WHERE user_id = ? ORDER BY created_at DESC (使用第一列，避免排序)

2.3 索引选择性分析

索引的选择性决定了索引的效果：

-- 计算索引选择性（越接近1越好） SELECT COUNT(DISTINCT column_name) / COUNT(*) AS selectivity, COUNT(*) AS total_rows FROM table_name; -- 示例：判断是否应该为gender列建索引 -- 如果选择性只有0.5（男/女），索引效果很差 -- 如果选择性是0.99（如email），索引效果很好
注意：选择性低于0.1的列通常不适合单独建索引。但对于复合索引，低选择性的列放在后面也可能有用。

2.4 索引覆盖优化

覆盖索引可以避免回表查询，大幅提升性能：

-- 原始查询：需要回表 SELECT id, name, email FROM users WHERE email = 'test@example.com'; -- 优化：创建覆盖索引 CREATE INDEX idx_user_email_cover ON users(email, id, name); -- 优化后：直接从索引获取数据，无需回表 -- EXPLAIN结果显示 Using index

第三章：查询优化实战

3.1 执行计划解读

理解EXPLAIN输出是优化的基本功：

字段 重要值 含义
type system > const > eq_ref > ref > range > index > ALL 访问类型，ALL最差
key 实际使用的索引名 NULL表示未使用索引
rows 预估扫描行数 越少越好
Extra Using index, Using where, Using filesort Using filesort需要优化

3.2 常见查询优化模式

模式1：避免SELECT *

-- 不好的做法 SELECT * FROM orders WHERE user_id = 123; -- 好的做法：只查需要的列 SELECT id, total_amount, status FROM orders WHERE user_id = 123;

模式2：优化分页查询

-- 传统分页：越往后越慢 SELECT * FROM orders ORDER BY id LIMIT 1000000, 20; -- 优化方案1：使用覆盖索引延迟关联 SELECT o.* FROM orders o INNER JOIN ( SELECT id FROM orders ORDER BY id LIMIT 1000000, 20 ) t ON o.id = t.id; -- 优化方案2：记住上一页的最大ID SELECT * FROM orders WHERE id > 1000000 ORDER BY id LIMIT 20;

模式3：优化OR条件

-- OR通常会导致索引失效 SELECT * FROM orders WHERE user_id = 123 OR status = 'pending'; -- 优化：使用UNION ALL SELECT * FROM orders WHERE user_id = 123 UNION ALL SELECT * FROM orders WHERE status = 'pending' AND user_id != 123;

模式4：优化LIKE查询

-- 前缀%会导致索引失效 SELECT * FROM users WHERE name LIKE '%张%'; -- 优化方案1：使用全文索引 SELECT * FROM users WHERE MATCH(name) AGAINST('张'); -- 优化方案2：使用覆盖索引 CREATE INDEX idx_user_name ON users(name(20)); SELECT id, name FROM users WHERE name LIKE '张%';

3.3 子查询优化

-- 不好的子查询 SELECT * FROM orders WHERE user_id IN (SELECT id FROM users WHERE status = 'active'); -- 优化：使用JOIN SELECT o.* FROM orders o INNER JOIN users u ON o.user_id = u.id WHERE u.status = 'active'; -- 或者使用EXISTS（当users表很大时） SELECT * FROM orders o WHERE EXISTS ( SELECT 1 FROM users u WHERE u.id = o.user_id AND u.status = 'active' );

第四章：数据库架构优化

4.1 表设计优化

字段类型选择

-- 整数类型选择 TINYINT -- 1字节，-128到127 SMALLINT -- 2字节，-32768到32767 INT -- 4字节，约±21亿 BIGINT -- 8字节，约±9.2×10^18 -- 字符串类型选择 CHAR(n) -- 固定长度，适合MD5、手机号 VARCHAR(n) -- 可变长度，适合姓名、邮箱 TEXT -- 长文本，避免在主表使用 -- 时间类型选择 DATETIME -- 8字节，范围1000-9999年 TIMESTAMP -- 4字节，范围1970-2038年 DATE -- 3字节，只存日期
设计原则：选择能够满足需求的最小数据类型。例如，状态字段用TINYINT而不是INT，可以节省75%的存储空间和内存。

范式与反范式

根据业务场景选择合适的范式级别：

-- 第三范式设计（减少冗余） CREATE TABLE orders ( id INT PRIMARY KEY, user_id INT, total_amount DECIMAL(10,2), created_at DATETIME ); CREATE TABLE order_items ( id INT PRIMARY KEY, order_id INT, product_id INT, quantity INT, price DECIMAL(10,2) ); -- 反范式设计（提升查询性能） CREATE TABLE order_summary ( id INT PRIMARY KEY, user_id INT, user_name VARCHAR(50), -- 冗余字段 total_amount DECIMAL(10,2), item_count INT, -- 冗余字段 created_at DATETIME );

4.2 分表分库策略

垂直拆分

按业务拆分不同的表到不同的库：

-- 用户库 user_db: users, user_profiles, user_settings -- 订单库 order_db: orders, order_items, order_logs -- 商品库 product_db: products, categories, inventory

水平拆分

按数据量拆分同一表到不同的库：

-- 按ID取模分表 order_0: id % 4 = 0 order_1: id % 4 = 1 order_2: id % 4 = 2 order_3: id % 4 = 3 -- 按时间范围分表 order_2024_01: 2024年1月的订单 order_2024_02: 2024年2月的订单 ...
分表时机：单表数据量超过500万行，或单表大小超过2GB时，应考虑分表。但分表会增加系统复杂度，需要权衡。

第五章：缓存策略

5.1 查询缓存

-- MySQL查询缓存（8.0已移除） -- 建议使用应用层缓存 -- Redis缓存示例 SET user:123:profile '{"name":"张三","email":"test@example.com"}' EXPIRE user:123:profile 3600 GET user:123:profile

5.2 缓存更新策略

策略 适用场景 优缺点
Cache Aside 读多写少 简单可靠，但可能有脏数据
Write Through 读写均衡 数据一致性好，写入延迟高
Write Behind 写多 写入快，但有数据丢失风险

5.3 缓存穿透/击穿/雪崩

-- 缓存穿透：查询不存在的数据 -- 解决方案：缓存空值或使用布隆过滤器 SET user:99999:exists "null" EX 300 -- 缓存击穿：热点key过期 -- 解决方案：设置永不过期或使用互斥锁 SETNX lock:user:123:profile 1 -- 缓存雪崩：大量key同时过期 -- 解决方案：随机过期时间 EXPIRE user:123:profile $((3600 + RANDOM % 600))

第六章：事务与锁优化

6.1 事务隔离级别

隔离级别 脏读 不可重复读 幻读 性能
Read Uncommitted 可能 可能 可能 最高
Read Committed 不会 可能 可能 高
Repeatable Read 不会 不会 可能 中
Serializable 不会 不会 不会 最低

6.2 锁优化策略

-- 减少锁持有时间 -- 不好的做法 BEGIN; SELECT * FROM orders WHERE id = 1 FOR UPDATE; -- 执行耗时操作... UPDATE orders SET status = 'completed' WHERE id = 1; COMMIT; -- 好的做法：把耗时操作移到事务外 SELECT * FROM orders WHERE id = 1; -- 获取数据 -- 执行耗时操作... BEGIN; UPDATE orders SET status = 'completed' WHERE id = 1; COMMIT; -- 使用乐观锁避免行锁 UPDATE products SET stock = stock - 1, version = version + 1 WHERE id = 123 AND version = 10;
死锁检测：定期检查死锁日志，发现并修复死锁问题。大多数数据库会自动检测并回滚死锁，但频繁的死锁会严重影响性能。

第七章：实战案例

案例：电商平台订单查询优化

问题描述：订单列表查询从200ms上升到5秒，用户投诉增加。

分析过程：

  1. 查看慢查询日志，发现SELECT * FROM orders WHERE user_id = ? ORDER BY created_at DESC LIMIT 20是主要慢查询
  2. EXPLAIN显示type=ref，Using filesort，扫描行数约50万
  3. 检查索引，只有idx_user_id(user_id)单列索引

优化方案：

-- 添加复合索引 CREATE INDEX idx_user_created ON orders(user_id, created_at DESC); -- 优化查询，只查需要的列 SELECT id, order_no, total_amount, status, created_at FROM orders WHERE user_id = ? ORDER BY created_at DESC LIMIT 20;

优化结果：

  • 查询时间从5秒降到50ms
  • 扫描行数从50万降到20
  • Extra从Using filesort变为Using index

总结：性能优化检查清单

优化项 检查点 预期效果
索引设计 是否覆盖高频查询？是否避免冗余索引？ 查询性能提升10-100倍
查询语句 是否避免SELECT *？分页是否优化？ 减少IO，提升响应
表结构 字段类型是否最优？是否有过多TEXT？ 减少存储和内存占用
缓存策略 热点数据是否缓存？缓存策略是否合理？ 减少数据库压力
事务优化 事务是否最小化？锁是否合理？ 提升并发能力
优化原则：先监控，后优化。不要盲目优化，每次优化都要有数据支撑。优化后要验证效果，并记录优化历史。
