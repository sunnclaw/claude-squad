RESTful API已成为现代Web服务的标准接口设计风格。一个设计良好的REST API不仅易于理解和使用，还能显著降低客户端与服务器之间的耦合度。本文将系统性地介绍RESTful API的核心设计原则，帮助你构建优雅、一致的API。

什么是REST？

REST（Representational State Transfer）是一种软件架构风格，由Roy Fielding在2000年的博士论文中提出。RESTful API是指遵循REST原则设计的Web API，具有以下核心特征：

  • 客户端-服务器分离：关注点分离，客户端负责用户界面，服务器负责数据存储和业务逻辑
  • 无状态：每个请求包含所有必要信息，服务器不保存客户端上下文
  • 可缓存：响应数据可以被缓存以提高性能
  • 统一接口：使用标准化的方法操作资源
  • 分层系统：客户端无需知道它连接的是最终服务器还是中间代理

资源命名规范

REST API的核心是资源。资源命名直接决定了API的清晰程度和专业性。

使用名词而非动词

URI应该表示资源，而不是动作。动作由HTTP方法表达。

正确示例： 
GET    /users          # 获取用户列表
GET    /users/123      # 获取ID为123的用户
POST   /users          # 创建新用户
PUT    /users/123      # 更新ID为123的用户
DELETE /users/123      # 删除ID为123的用户
错误示例： 
GET /getUsers
POST /createUser
POST /deleteUser/123

使用复数形式

资源名称统一使用复数形式，保持一致性。

# 推荐：复数形式 /users /products /orders # 不推荐：单数形式 /user /product /order

表达层级关系

当资源之间存在从属关系时，通过嵌套URI表达。

# 获取用户的所有订单 GET /users/123/orders # 获取用户的特定订单 GET /users/123/orders/456 # 注意：嵌套层级一般不超过2-3层 # 如果关系过于复杂，考虑使用查询参数 GET /orders?userId=123

使用小写字母和连字符

URI应使用小写字母，多个单词之间用连字符连接。

正确： 
/user-profiles
/order-items
/product-categories
错误： 
/UserProfiles
/orderItems
/product_categories

HTTP方法的正确使用

HTTP方法定义了对资源的操作类型。正确使用HTTP方法是RESTful API的基础。

方法 操作 幂等性 安全性
GET 获取资源 是 是
POST 创建资源 否 否
PUT 完整更新资源 是 否
PATCH 部分更新资源 否 否
DELETE 删除资源 是 否
幂等性说明：幂等操作是指执行一次和执行多次效果相同。GET、PUT、DELETE是幂等的，POST和PATCH不是。这在重试机制设计中非常重要。

GET方法

GET用于获取资源，应该是安全且幂等的。不应该通过GET修改资源状态。

# 获取用户列表 GET /users # 获取单个用户 GET /users/123 # 带查询参数的过滤 GET /users?status=active&role=admin

POST方法

POST用于创建新资源。请求体包含资源数据，响应应返回新创建资源的URI。

# 创建新用户 POST /users Content-Type: application/json { "name": "张三", "email": "zhangsan@example.com" } # 响应 HTTP/1.1 201 Created Location: /users/124 { "id": 124, "name": "张三", "email": "zhangsan@example.com", "createdAt": "2026-03-10T10:00:00Z" }

PUT vs PATCH

PUT用于完整替换资源，PATCH用于部分更新。

PUT示例 - 完整更新： 
PUT /users/123
{
  "name": "李四",        # 必须提供所有字段
  "email": "lisi@example.com"
}
PATCH示例 - 部分更新： 
PATCH /users/123
{
  "email": "newemail@example.com"  # 只更新email字段
}

HTTP状态码规范

正确使用HTTP状态码是API专业性的重要体现。状态码应该准确反映请求的处理结果。

常用状态码分类

范围 类别 说明
1xx 信息性 请求已被接收，正在处理
2xx 成功 请求已成功处理
3xx 重定向 需要进一步操作
4xx 客户端错误 请求有语法错误或无法完成
5xx 服务器错误 服务器处理请求时出错

最常用的状态码

成功响应（2xx）

200 OK # 请求成功（GET、PUT、PATCH） 201 Created # 资源创建成功（POST） 204 No Content # 成功但无返回内容（DELETE）

客户端错误（4xx）

400 Bad Request # 请求格式错误或参数无效 401 Unauthorized # 未认证 403 Forbidden # 已认证但无权限 404 Not Found # 资源不存在 409 Conflict # 请求冲突（如重复创建） 422 Unprocessable Entity # 语义错误（验证失败） 429 Too Many Requests # 请求过于频繁

服务器错误（5xx）

500 Internal Server Error # 服务器内部错误 502 Bad Gateway # 网关错误 503 Service Unavailable # 服务暂时不可用
注意事项：不要将业务错误映射为5xx状态码。如果用户余额不足，应返回4xx状态码配合错误详情，而非500。

错误处理规范

当API返回错误时，应提供一致的错误响应格式，帮助客户端开发者快速定位问题。

推荐的错误响应格式

{ "error": { "code": "VALIDATION_ERROR", "message": "请求参数验证失败", "details": [ { "field": "email", "message": "邮箱格式不正确" }, { "field": "password", "message": "密码长度至少8位" } ], "requestId": "req-abc123", "timestamp": "2026-03-10T10:30:00Z" } }

错误码设计原则

  • 使用大写字母和下划线：VALIDATION_ERROR、RESOURCE_NOT_FOUND
  • 语义明确：错误码应能直接反映错误类型
  • 便于程序处理：客户端可以根据错误码进行分类处理
  • 提供调试信息：包含requestId便于日志追踪

版本控制

API不可避免会发生变化，版本控制策略决定了变化的平滑程度。

常见的版本控制方式

1. URI路径版本（推荐）

# 最常用的方式，直观明了 GET /api/v1/users GET /api/v2/users

2. 查询参数版本

# 灵活但不够直观 GET /api/users?version=1 GET /api/users?version=2

3. 请求头版本

# 保持URI简洁，但不易发现 GET /api/users Accept: application/vnd.myapi.v1+json
最佳实践：推荐使用URI路径版本，因为它直观、易于测试、便于缓存。同时支持至少两个主版本，给客户端足够的迁移时间。

分页与过滤

当资源数量庞大时，分页和过滤是必不可少的API特性。

分页设计

# 基于偏移量的分页 GET /users?page=2&limit=20 # 基于游标的分页（大数据量推荐） GET /users?cursor=eyJpZCI6MTAwfQ&limit=20

分页响应格式

{ "data": [...], "pagination": { "page": 2, "limit": 20, "total": 150, "totalPages": 8, "hasNext": true, "hasPrev": true } }

过滤与排序

# 过滤 GET /users?status=active&role=admin # 搜索 GET /users?search=zhang # 排序 GET /users?sort=createdAt&order=desc # 字段选择 GET /users?fields=id,name,email

认证与授权

REST API通常需要认证来识别用户身份，授权来控制访问权限。

常见的认证方式

Bearer Token（推荐）

# 请求头携带Token GET /api/v1/users Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

API Key

# 通过请求头 X-API-Key: your-api-key # 或查询参数（不推荐，有安全风险） GET /api/v1/users?api_key=your-api-key
安全提醒：永远不要在URI中传递敏感信息（如Token、密码），因为URI会被记录在服务器日志、浏览器历史记录中。

HATEOAS：超媒体驱动

HATEOAS（Hypermedia as the Engine of Application State）是REST成熟度模型中的最高级别，意味着API响应包含相关操作的链接。

{ "id": 123, "name": "张三", "email": "zhangsan@example.com", "_links": { "self": { "href": "/users/123" }, "orders": { "href": "/users/123/orders" }, "update": { "href": "/users/123", "method": "PUT" }, "delete": { "href": "/users/123", "method": "DELETE" } } }
实用性说明：HATEOAS在实际项目中使用较少，但对于需要高度动态性的公开API很有价值。大多数内部API可以简化这一层。

API设计检查清单

完成API设计后，使用这个检查清单验证：

检查项 标准
URI命名 使用名词复数、小写、连字符
HTTP方法 符合语义，GET不修改状态
状态码 准确反映处理结果
错误响应 格式统一，信息完整
版本控制 策略明确，向后兼容
认证授权 安全机制完善
文档 API文档完整可访问

总结

RESTful API设计不仅仅是技术问题，更是关于如何用一致、可预测的方式暴露业务能力。核心原则可以总结为：

  • 资源为中心：URI表示资源，HTTP方法表示操作
  • 语义清晰：正确使用HTTP方法和状态码
  • 一致性：命名、格式、错误处理保持统一
  • 可演进：通过版本控制支持API迭代
  • 开发者友好：文档完善，错误信息明确
