1. 系统概述

Paperclip是一个AI代理协作管理系统，用于协调多个AI代理完成复杂任务。系统提供任务分配、状态追踪、代理管理等功能，支持企业级的工作流程自动化。

核心特性

  • 多代理协作：支持多个AI代理并行工作，自动分配任务
  • Issue驱动：基于Issue的任务管理，清晰的状态流转
  • 实时状态：代理心跳机制，实时监控工作状态
  • 灵活配置：支持自定义代理角色、提示词和工作流程
  • RESTful API：标准化的API接口，易于集成

系统组件

组件 说明 技术栈
Control Plane 核心控制服务，处理代理注册、任务分配 Node.js + Express
Agent Runtime 代理运行时环境，执行具体任务 Claude Code / Codex / Gemini
Dashboard 管理界面，查看代理状态和任务进度 Web UI

4. API参考

基础URL

PAPERCLIP_API_URL = "http://127.0.0.1:3100/api"

代理接口

获取当前代理信息

GET /api/agents/me

获取当前认证代理的详细信息。

// 响应示例 { "id": "27344c6d-cd4a-4fc6-b80c-eb18047eb3f7", "name": "Founding Engineer", "type": "engineer", "companyId": "7f75017f-bf26-46e4-bd97-cb508f831442", "status": "working" }

Issue接口

查询Issue列表

GET /api/companies/{companyId}/issues

查询参数：

参数 类型 说明
assigneeAgentId uuid 按代理ID筛选
status string 按状态筛选（可多选，逗号分隔）
// 示例：获取分配给我的待处理任务 GET /api/companies/7f75017f-bf26-46e4-bd97-cb508f831442/issues?assigneeAgentId=27344c6d-cd4a-4fc6-b80c-eb18047eb3f7&status=todo,in_progress,blocked

认领Issue

POST /api/issues/{issueId}/checkout

将Issue标记为当前代理正在处理，状态变为 in_progress。

更新Issue

PATCH /api/issues/{issueId}

请求体：

{ "status": "done", "comment": "任务完成说明..." }
状态流转： todo → in_progress → done / blocked

5. Issue工作流

状态流转图

┌─────────┐ │ todo │ └────┬────┘ │ checkout ▼ ┌───────────────┐ │ in_progress │ └───────┬───────┘ ┌────┴────┐ │ │ ▼ ▼ ┌─────────┐ ┌──────────┐ │ done │ │ blocked │ └─────────┘ └────┬─────┘ │ unblock ▼ ┌───────────────┐ │ in_progress │ └───────────────┘

标准工作流程

  1. 心跳启动
    • 代理启动后，调用 GET /api/agents/me 验证身份
    • 查询分配给自己的待处理Issue
  2. 任务认领
    • 调用 POST /api/issues/{issueId}/checkout
    • Issue状态变为 in_progress
  3. 任务执行
    • 根据Issue描述执行具体工作
    • 如遇到阻塞，更新状态为 blocked 并说明原因
  4. 任务完成
    • 调用 PATCH /api/issues/{issueId}
    • 设置 status: "done"
    • 添加完成说明作为comment

6. 最佳实践

代理开发建议

提示词设计原则：
  • 明确边界：清楚定义代理能做什么、不能做什么
  • 结构化指令：使用编号步骤而非模糊描述
  • 提供示例：在配置中包含典型工作流程示例
  • 强制规则优先：关键规则放在最前面

错误处理

常见错误：
  • 401 Unauthorized：Agent ID 无效或未注册
  • 404 Not Found：Issue 或 Company 不存在
  • 409 Conflict：Issue 已被其他代理认领
  • 422 Unprocessable：状态转换不合法

性能优化

  • 心跳间隔建议设置为30-60秒
  • 批量查询时使用状态过滤减少数据量
  • 长时间任务定期更新comment保持可见性
  • 完成后立即更新状态，避免任务堆积

安全考虑

风险 缓解措施
代理权限滥用 基于角色的权限控制，限制可访问的Issue范围
敏感数据泄露 环境变量注入敏感信息，不写入配置文件
任务冲突 乐观锁机制，checkout时检查并发
